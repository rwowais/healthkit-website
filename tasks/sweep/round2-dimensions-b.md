## 1. [MEDIUM] (perf) performance
**Cloud mode: one tab refocus triggers up to 10 full cloud loads (full state doc + entire logs table each) because every useAppState consumer is an independent instance with its own focus/visibility sync listeners**
- Area: src/hooks/useAppState.ts:161-202; src/lib/datasource.ts:728-825
- Evidence: useAppState is a per-component hook, not shared context. On /today five instances mount simultaneously: src/app/today/page.tsx:201, src/components/Shell.tsx:44, src/components/Reminders.tsx:45 (rendered by Shell:60), src/components/TimezoneSentry.tsx:30 (layout.tsx:174), src/components/today/QuickAdd.tsx:24 (today/page.tsx:3003). Each instance registers sync() on window 'focus' AND document 'visibilitychange' (useAppState.ts:194-195) with no document.hidden guard — both events fire on a tab refocus, and visibilitychange fires again on tab HIDE, so a single away-and-back cycle runs sync() up to 15 times (5 on hide + 10 on return). Each sync() calls activeDataSource.load() unconditionally (useAppState.ts:169); in cloud mode SupabaseDataSource.load() does a SELECT of the full state document (datasource.ts:736-740), writes it whole to localStorage (783-786), re-parses+normalizes it (788), then reconcileLogs() SELECTs EVERY row of protocolize_logs with no date filter (readLogDays, datasource.ts:592-595: .select("log").eq("user_id", userId)) and writes the full state to localStorage a second time (saveState at 666). Probe-measured: a 1-year account's state JSON is 551 KB and its logs table another ~550 KB (365 rows × ~1.5 KB) — so one refocus ≈ 20+ Supabase queries and ~10 MB transferred, growing linearly with account age. The same fan-out also runs on every initial page load (5 mount loads, useAppState.ts:85-103) and after every save flush via STATE_EVENT (4 sibling instances each do a full cloud load, ~4 MB per toggle burst). The hook's own comment (177-187) shows the team fixed the cross-TAB storage-event storm by disabling that listener in cloud mode, but the same-tab ×5 instance amplification and the on-hide sync were left in place. Not on tasks/sweep/triage.txt (no performance entries; #74 covers cross-tab UX divergence only).
- Why: On a mobile PWA (the primary form factor) every app-switch burns megabytes of cellular data and battery, multiplies Supabase read/egress costs ~10× for zero benefit (all 5 loads return identical data and 4 of the 5 stableStringify comparisons discard the result), and the redundant localStorage writes (2 per load × 10 loads) block the main thread right at the resume moment when the user wants to tap.
- Fix: Hoist state into a single React context provider (one load, one set of listeners, all consumers subscribe to the same state object), or as a minimal patch: (a) skip sync() when document.visibilityState === 'hidden', (b) coalesce concurrent loads behind one module-level in-flight promise in SupabaseDataSource.load(), (c) debounce focus+visibilitychange into one sync per resume.

## 2. [MEDIUM] (perf) performance
**Unbounded full-document persistence: state doc grows ~1.5 KB/day forever and the WHOLE document (551 KB at 1 year, 1.05 MB at 2 years) is upserted to Supabase on every toggle burst and re-downloaded with the full logs table on every load**

Not on triage (no perf entries; the per-day table work #40 explicitly left 'document stays authoritative' as a known phase, but the read path's full-table scan and the absence of any doc-side windowing are unflagged).

- Fix: Finish Phase 3 of the logs split: strip dailyLogs older than a window (e.g. 60-90 days) out of the state document and make the per-day table authoritative for history, with date-filtered reads (.gte('log_date', cutoff)) and on-demand fetch for Insights' long-range views; until then, at minimum add a date filter to readLogDays and skip the doc upsert when only a dailyLog changed (the per-day upsert already covers it).

## 4. [MEDIUM] (security) security-injection
**Share-target route opens unvalidated `url` param via window.open — javascript:/data: scheme reaches the click sink**
- Area: src/app/share/page.tsx:55-58 + 110-116
- Evidence: /share is registered as a Web Share Target in manifest.ts (action:"/share", lines 106-125), so other apps populate title/text/url as GET params. share/page.tsx reads `url = search.get("url")` (line 47). `looksLikeUrl = !!url || /^https?:\/\//.test(...)` (55-58) makes ANY non-empty url render the "Open the link" button. openUrlExternally (110-116) builds `const target = url || ...` and calls `window.open(target, "_blank", "noopener,noreferrer")` with NO scheme check — url="javascript:..." or a data: URL flows straight to window.open. Not present in triage.txt (no /share or window.open entry).
- Fix: Reuse safeUrl() (or inline `new URL(target)` requiring protocol http:/https:) inside openUrlExternally before window.open, and compute looksLikeUrl / show the "Open the link" button only for a sanitized http(s) value. A non-http url should be treated as plain text, never opened.

## 6. [MEDIUM] (bug) cms-admin-publish

## 13. [LOW] (inconsistency) data-lifecycle
**scrubBiomarkers rewrites a legitimate reading's date on any clock-behind device and the mutation syncs cloud-wide — the reader-guard principle applied to logs was not applied to biomarker dates**
- Area: src/lib/storage.ts:94-104 (scrubBiomarkers date clamp) vs storage.ts:85-92 (round-2 note: future LOGS are kept verbatim, readers guard) ; sync-up path src/lib/datasource.ts:783-794
- Evidence: scrubBiomarkers (storage.ts:99) does `b.date > today ? { ...raw, date: today }` where today = getDateString(undefined, getTz(s.settings)) — same tz on all devices (settings.timezone is synced state), so 'today' diverges only via the device clock. A reading logged today on device A is 'future-dated' to device B whose clock is a day behind; B's normalize() (runs on EVERY load including cloud reads, storage.ts:462) rewrites its date to B's yesterday, and the mutated list is persisted by the load path (datasource.ts:783-786 writes the chosen state to localStorage; 794 pushes a dirty merge up) — permanent, account-wide. The same audit round explicitly REMOVED this pattern for daily logs: the comment at storage.ts:85-92 records that 'an earlier normalize-time clamp destroyed that day's completions in exactly the traveler/multi-device case it claimed to protect' and concludes 'Readers guard; data is never mutated' — but the biomarker date clamp two lines below still mutates at the load boundary, with no reader-side alternative (latestBiomarker at storage.ts:1414-1421 just sorts by date with no today guard).

Known issues #39/#49 covered the missing floor/scrub-hole (since fixed); this is the surviving asymmetry in the fix itself: mutation-on-load for biomarker dates vs the reader-guard doctrine adopted for logs the same day.
- Fix: Apply the logs doctrine to biomarker dates: keep future-dated readings verbatim in scrubBiomarkers (only drop implausible VALUES), and guard the readers instead — latestBiomarker and the band/forecast/engine consumers ignore entries with date > today, exactly like calculateStreak ignores future logs. The reading then self-resolves when the slow clock catches up instead of being permanently rewritten.
