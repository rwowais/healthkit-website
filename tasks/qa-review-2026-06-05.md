# Comprehensive QA Review — 2026-06-05

104 agents · 86 findings → **70 confirmed** after adversarial verification (4 P0 · 21 P1 · 29 P2 · 16 P3).

## Overall health
Solid and genuinely well-built for a solo-founder pre-launch product, but with a cluster of real correctness bugs concentrated in exactly the two areas the project flagged as recently-changed: cloud sync / data-merge, and constraint-aware scheduling. The architecture is sound (clean engine pipeline, single AppState, token-driven theming, own-row RLS) and the skeptic pass already filtered noise, so almost every remaining finding is a true positive with a precise file:line and a small surgical fix. The dangerous edge is the merge/reset path in datasource.ts plus storage.ts: several findings independently converge on the same structural weakness (mergeStates drops whole state slices, reconcileLogs ignores recency, clearAllData leaves stale sync markers) which together can silently lose user data across devices. None fire on every load; they need specific timing/sign-out/multi-device sequences, so the app behaves correctly in the common case, which is exactly why these bugs survived. The design is strong and on-brand (calm, advisory; the onboarding merge moment is a standout), with two honest weaknesses: the Today morning surface stacks too many advisory cards before the actual checklist, and the premium intelligence layer makes a few claims (causal verdicts, 1-point-higher, keystone-on-flat-data) that overreach the data and undercut the trust the product stakes its credibility on. No security holes of note (entitlements are client-side by deliberate design; RLS is own-row). Net: not fragile, but not launch-clean either; the data-safety cluster must be closed before enabling cloud sync at scale.

## Recommendation (fix order)
Fix the data-safety cluster FIRST, before anything else and before turning cloud sync on for real users; these are the only findings that silently and permanently lose user data. In order: (1) reconcileLogs recency merge at datasource.ts:586-596, (2) mergeStates dropped slices at datasource.ts:421-490, (3) clearAllData stale sync markers (pz:pending-sync and pz:recon) at storage.ts:463-471. All three are small localized edits in two files and together close every silent cross-device data-loss path. Then fix the safetyFlags reset drop at profile/page.tsx:1031-1050; one line, safety-critical. Then knock out the constraint-aware-scheduling cluster as a batch (publish timeWindow backfill, Today move-menu hard-window guard, Reminders pipeline parity, edit-mode-on-past-day reset) since they share the recently-changed code and several are one-liners. The intelligence-layer honesty fixes (zero-variance, delta floor, experiment verdict copy) and the Today morning-card consolidation are real and worth doing but are credibility/polish, not data-loss; schedule them after the integrity work. Do NOT enable Stripe until a fulfillment path sets tier=premium; it is a launch-blocker tied to that switch, not a redeploy.

## Design opinions
- Pull the Today morning surface toward a strict one-hero rule: greeting plus ONE framing card (MorningBriefing OR Operating Summary, never both because they say the same thing twice) plus Up Next plus the timeline. Fold WeeklyGoal/signal-chips into the hero and collapse the check-in plus QuickLog into one compact strip so the checklist is not pushed below the fold.
- Hold the premium intelligence layer to the honesty bar it advertises (measured from your own check-ins): no causal verdicts on uncontrolled n=4 windows, no keystone/Proven-by-your-data claims on near-constant data, no runs-1-point-higher when the real gap rounds to 0, no single-dimension attribution on tiny samples. The numbers must earn the first-person framing.
- Keep engagement/streak/share stats on LIVE state and only the correlation/keystone intelligence on the free-tier 3-day delay. Gating a personal best, a streak card, a monthly-report share image, or a consistency calendar behind the delay produces wrong, contradictory, and sometimes shareable-wrong numbers, and silently blacks out the most motivating habit-loop cards.
- The LIGHT theme is the default and is the weakest-tested surface: a couple of structural elements leak hardcoded white (toggle knobs, the Today connector rail) and disappear, and the text-4 token fails WCAG AA in both themes for real content. Treat looks-right-and-passes-AA-in-light-mode as a release gate, and build one shared Toggle primitive instead of ~10 hand-rolled switches.
- Protect the genuinely excellent patterns as you trim: the onboarding Your-day-assembled moment makes the core differentiator tangible at the right time and is backed by real de-dup math, the calm/advisory voice is consistent and on-brand, and the per-behavior here-is-the-science-behind-this-timing on moves is a standout trust mechanic. Pull the rest of the app toward that restraint rather than adding cards.
- The landing page never shows the product or defines protocol in 10 seconds; the differentiator (expert routines merged into one adaptive de-duplicated day) is invisible until after 8 onboarding steps. Add one concrete proof above the fold and make the onboarding name field skippable so curious visitors are not blocked at the very first gate.

### P0 (4)
**mergeStates drops local supplements, supplementMeta, protocols & insights on the dirty-sync merge**  
_where:_ src/lib/datasource.ts:421-491 (mergeStates return object)  
_why:_ The merge return is { ...cloud, settings, dailyLogs, biomarkers, customPacks, installedPacks, pausedPacks, behaviorOverrides }. It never merges or even spreads local's supplements / supplementMeta / protocols / insights — those come ONLY from `...cloud`. On the dirty-load path (chooseCloudLoad → mergeStates), any un-pushed local edit to the user's supplement stack (add/remove/dose/inventory via /supplements), supplement meta, or legacy protocol pillars is silently discarded in favor of the cloud copy, even though the whole point of that path is to preserve un-pushed local edits.  
_fix:_ In mergeStates, explicitly reconcile the missing slices: supplements (by-id union preferring local on collision, like biomarkers/customPacks), supplementMeta ({...cloud, ...local}), insights (by-id union), and protocols (prefer local). At minimum spread local's supplements/supplementMeta so un-pushed stack edits survive.  
_reviewer:_ storage-sync

**reconcileLogs overwrites field-merged day logs with stale per-day cloud rows (defeats mergeDailyLog)**  
_where:_ src/lib/datasource.ts:586-593 (reconcileLogs) vs load() dirty path 723-740  
_why:_ On a dirty load, load() builds a local-preferring document via mergeStates (which runs mergeDailyLog per day), writes+normalizes it (norm), then calls reconcileLogs(sb, userId, norm). reconcileLogs reads protocolize_logs rows and does `byDate.set(d.date, d)` with the cloud ROW winning verbatim for every date present in the table; only doc-days absent from the table are kept (591-592). There is NO mergeDailyLog and NO updatedAt recency here. So for any day that has both a (stale) per-day row and a freshly-merged local edit, the carefully field-merged log is thrown away and replaced by the stale row — exactly the days where conflict resolution mattered. Then result is saved back to localStorage, persisting the loss.  
_fix:_ In reconcileLogs, when a date exists in both the table row and base.dailyLogs, reconcile via mergeDailyLog(row, docDay) (doc/local as the more-recent-intent `b`) instead of letting the row win unconditionally; or compare updatedAt and keep the newer. Keep the absent-day backfill as-is.

### P1 (21)
**Bed-anchored HARD windows wrap (lo>hi) for post-midnight bedtimes → window silently unenforced + all block pills disabled in editor**  
_where:_ src/lib/time.ts:207-258 (resolveTimeWindow/isWithinWindow/clampToWindow/windowBlocks) + src/lib/engine.ts:360-364 (HARD heal) + src/components/BehaviorSheet.tsx:105-115,180  
_why:_ For anchor=bed windows like wind-down/screens-off/dim-lights ({min:-240..-180,max:0}), resolveTimeWindow does bed+=1440 when bedtime<=wake, then norm()s lo and hi. With a post-midnight bedtime (e.g. 00:30) lo normalizes into the prior evening (~20:30) and hi wraps to ~00:30, so lo>hi. Every window helper then bails via the `w.lo > w.hi` guard: isWithinWindow always returns true, clampToWindow returns the value unchanged, and windowBlocks returns []. Two consequences: (a) the engine's strict-window heal (engine.ts:360) never clamps, so a circadian HARD behavior is no longer kept in its window for night-owls; (b) BehaviorSheet computes allowedBlocks = new Set([]) so `forbidden` is true for morning/afternoon/evening/anytime — every block pill is disabled and the user cannot move the behavior by block at all.  
_fix:_ Handle the wrapped window instead of disabling it. In resolveTimeWindow keep lo/hi on a single monotonic [wake, wake+1440) axis (don't norm() both independently — or return a wrapped flag), and make isWithinWindow/clampToWindow/windowBlocks treat lo>hi as a real wrapping interval (m>=lo || m<=hi) rather than 'no window'. windowBlocks should still enumerate the spanned blocks across the wrap so the editor enables the correct pills.

**Zero-variance bypass lets keystone / "Proven by your data" assert max confidence on near-constant data**  
_where:_ src/lib/intel.ts:256 (keystone: `const d = pooledSD > 0 ? (mD - mN) / pooledSD : 99;`) and src/lib/intel.ts:336 (whatWorks/cohenD: `sd > 0 ? (mA - mB) / sd : mA > mB ? 99 : 0`)  
_why:_ When within-group variance is 0 but the two group means differ, pooledSD/sd is 0 and the effect size is hard-coded to 99 — which clears every multiple-comparison dThreshold (0.35–0.55). The whole point of these gates is to refuse a causal-sounding claim without a real effect size; the degenerate branch silently skips it. This promotes a keystone / first-person "your data shows X works for you" claim on essentially constant data.  
_fix:_ Treat zero pooled SD as "no measurable effect", not infinite: `const d = pooledSD > 0 ? (mD - mN) / pooledSD : 0;` (and the cohenD helper return 0 when sd===0). A real lift will have nonzero variance.
