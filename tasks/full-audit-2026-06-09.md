### [data-risk] (data-sync-auth-wall) deleteAccount destroys cloud data BEFORE the auth-user delete, then reports success even if that delete fails — user is told 'deleted' but can still log back in to an empty account
- **Evidence:** auth.ts:129-165: deleteAccount() first hard-deletes protocolize_state, protocolize_logs, push_subscriptions, cms_admins for the user (lines 136-147), THEN calls rpc('delete_my_account') (line 152). Your sign-in record will be removed shortly…' } (lines 153-162). profile/page.tsx:975-988 treats ok:true as success: it calls clearAllData() and shows a toast, then routes to '/'. The auth user still exists and can sign in again — but its data rows are already gone, so they return to an empty, re-onboarding account while believing the account was fully deleted.
- **Why:** Irreversibly deleting the data rows before confirming the identity delete inverts the safe order.
